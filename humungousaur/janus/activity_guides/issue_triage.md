# Activity Skill Pack: Issue Triage

## Summary

Use when the user is reading, sorting, labeling, prioritizing, assigning,
updating, or following up on tracked work items. Optimize for queue clarity,
blocker detection, and safe status continuity.

## Signals

- Issue, task, ticket, backlog, board, label, assignee, milestone, status,
  comment, or priority activity.
- Repeated switching between an issue, code, docs, messages, incidents, or
  planning artifacts.
- Creation, closure, reopen, assignment, escalation, duplicate marking, or
  status-change events.

## Helpful Moments

- A queue is reviewed in batches and a concise priority summary may help.
- An item appears blocked, stale, duplicated, escalated, or ready for handoff.
- The user returns to an issue after a gap or after related code/debugging work.
- The user explicitly asks for triage, prioritization, response drafting, or next
  actions.

## Stay Silent When

- Board updates are background sync, automation, or notification noise.
- Assistance would require reading private issue bodies, comments, customer data,
  names, URLs, attachments, or code without approval.
- The project or item is muted, private, policy-blocked, or marked no-assistance.
- The user is rapidly scanning a queue without a safe boundary.

## Deep Dive Triggers

- Reading issue descriptions, comments, attachments, linked PRs, logs, customer
  reports, or related documents.
- Drafting public responses, changing labels/status/assignees, closing issues, or
  creating follow-up work.
- Comparing duplicates or prioritizing from private content.

## Memory Guidance

- Store redacted issue/project hashes, broad work type, status category, priority
  category, blocker state, handoff state, and explicit user decisions.
- Remember safe links between issues and related work artifacts using hashes.
- Do not retain customer details, issue text, comments, attachments, names, or
  exact titles without approval.

## Privacy Notes

- Treat issue trackers as mixed public/private workspaces.
- Ask before reading content or mutating tracked work.
- Prefer queue-level summaries and redacted state over raw issue details.
