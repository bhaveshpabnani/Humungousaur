# Governance

Humungousaur is currently maintained by the project owner and invited maintainers.

## Decision Principles

- User safety and local trust come before automation breadth.
- Real behavior proof beats demos, mocks, and claims.
- Cross-platform parity matters for public desktop features.
- New capabilities should usually begin as skills or explicit tool contracts, not hidden natural-language routing.
- External projects are reference material unless license, provenance, and integration boundaries are reviewed.

## Maintainer Responsibilities

Maintainers are responsible for:

- triaging issues and pull requests
- keeping release gates honest
- reviewing security-sensitive surfaces
- protecting approval and audit boundaries
- deciding whether a capability belongs in core, a bundled skill, an optional integration, or a third-party extension

## Contribution Path

Small bug fixes and docs corrections can go straight to pull request. Larger features, new external integrations, new desktop behavior, and security-sensitive changes should start with an issue so scope and safety expectations are clear before implementation.

## Release Authority

Public releases should not be tagged or announced until the release runbook passes or the remaining blockers are explicitly documented in the release notes.
